# ToneAI NUX — tone-accuracy audit findings

**Date:** 2026-07-18
**Method:** read-only multi-agent workflow audit (43 subagents, one per failure-mode,
each finding adversarially verified by an independent skeptic), plus a ground-truth
pass against the open-source [tuntorius/mightier_amp](https://github.com/tuntorius/mightier_amp)
device definitions (MIT) — the same source this project's NUX support was originally
built from.
**Outcome:** diagnosis only. No code was changed. A proposed `nux-qr-tool` validation
PR was written and then **withdrawn** (see "Why no fix shipped").

---

## Trigger

Reports that generated NUX tones are "often inaccurate or wrong."

## What the audit found (11 findings → 6 ranked, adversarially verified)

1. **No vendored ID/parameter catalog.** The imprint carries no amp/cab/effect
   `id → model` tables or effect-parameter (p1/p2/p3) semantics. The model picks
   device-specific IDs from its own knowledge, and the encoder writes them through.
2. **No independent correctness oracle.** The only in-repo "expected" payload is
   self-generated by the encoder under test — circular. Nothing validates output
   against a real device.
3. **`coerceParams` failed open on a missing device** — silently defaulted to
   `plugpro` (Pro / 113-byte), emitting a Mighty Plug Pro payload for whatever
   hardware the caller actually had.
4. **Pickup calibration is prompt-only.** The per-pickup gain/gate/EQ deltas in
   `IDENTITY.md` are persona guidance, not enforced in code.
5. **Orphaned self-test fixture** — claims regeneration by a build script that
   doesn't exist; unreferenced anywhere.
6. **(Positive) The encode → QR → decode round-trip is lossless.** The byte-packing
   and QR layer are sound. Whatever is wrong is a *data* problem, not an encoding bug.

**Gaps the audit itself missed** (named by its own completeness critic): effect/amp
signal-chain **ordering**, and amp/cabinet-IR **pairing coherence** (a valid-but-wrong
cab passes every ID check).

## Ground-truth reality check — the important caveat

When the fix was scoped, the amp/cab IDs were checked against the mightier_amp source.
**The IDs that could be checked were correct** (e.g. Plug Pro amp id 5 = Twin Reverb,
cabinet id 2 = DR112 — both matched the imprint's own labels). Combined with the fact
that this project's encoder lineage was **built from mightier_amp in the first place**,
the honest conclusion is:

> The IDs were most likely already right. The audit surfaced a real *structural*
> weakness — design-by-model-memory with no ground-truth backstop — but it never
> **proved** that wrong IDs are the cause of the reported "inaccurate tones."

The reported inaccuracy could equally be gain staging, effect parameters, signal-chain
order, or musical/amp-choice quality — none of which an ID catalog addresses.

## Why no fix shipped

A `nux-qr-tool` PR was written (vendored catalog + reject-invalid-ID validation +
the required-device fix) and then withdrawn, because:

- It was **almost entirely additive** — a large new validation surface built on an
  **unproven** root cause.
- The spot-checked IDs were correct and the encoder descends from the same source we'd
  validate against, so the catalog would largely be a **safety net, not a bug fix**.
- Shipping it risked presenting "we fixed the tones" when the actual cause was never
  isolated.

The one change that stands on its own merit regardless of root cause is #3 (make
`device` required instead of silently defaulting to `plugpro`) — a genuine correctness
fix, small and non-additive. Worth doing in isolation if/when the encoder is touched.

## Recommended next step (before any fix)

**Isolate encoder-vs-model with an oracle we already trust.** `mighty-ai-qr-web` was
built and validated by the author. Generate the *same* preset intent through both
`nux-qr-tool` and `mighty-ai-qr-web` and diff the payloads:

- **If they match** → the encoder and IDs are fine; the "inaccurate tones" live in the
  model's musical/gain/parameter choices, and effort should go there, not into
  validation.
- **If they differ** → a real encoder discrepancy exists; then (and only then) a catalog
  and validation earn their place.

For absolute certainty, capture a few known-good QR exports from a physical NUX device
via the real MightyAmp app and diff against them — the only true correctness oracle.

## Status

Diagnosis recorded; no code changed. Branches/PR from the abandoned fix attempt were
deleted. This document is the durable record of what the audit established and, just as
importantly, what it did **not** establish.
